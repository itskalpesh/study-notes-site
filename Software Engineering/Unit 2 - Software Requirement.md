# Unit II: Software Requirements
### Software Requirement :
A software requirement is a description of the functions, features, constraints, and quality attributes that a software system must satisfy.

Simple Definition: Software requirements describe what the software should do and how it should perform.

### Importance of Software Requirements

- Helps understand customer needs.
- Provides a clear understanding of the project.
- Reduces misunderstandings.
- Helps estimate project cost and time.
- Serves as the foundation for design, coding, and testing.
- Improves software quality.

### Characteristics of Good Software Requirements
- Clear and easy to understand.
- Complete and accurate.
- Consistent.
- Verifiable (can be tested).
- Feasible (can be implemented).
- Traceable.
- Easy to modify.

---

### 1. Functional Requirements
- Definition:
    Functional requirements describe what the software system should do. They specify the services, functions, and operations the system must perform.

- Characteristics 
    - Describe system functionality.
    - Define inputs and outputs.
    - Specify business rules.
    - Can be tested directly.
    - Focus on user requirements.

- Examples
    - User registration and login.
    - Online payment processing.
    - Generate reports.
    - Search for products.
    - Reset password.

- Advantages
    - Clearly defines system functions.
    - Helps developers understand customer needs.
    - Simplifies software testing.
    - Improves software quality

---
### 2. Non-Functional Requirements
- Definition:
    Non-functional requirements describe how the software should perform. They specify the quality attributes and performance standards of the system.
- Types
    - Performance  
        - Fast response time.
        - High processing speed.
        - Example: Website should load within 3 seconds.
    - Security 
        - Protect data from unauthorized access.
        - User authentication and encryption.
        - Example: Login using username and password.
    - Reliability 
        - Software works without failure.
        - High availability.
        - Example: Banking software should work 24×7.
    - Usability 
        - Easy to use.
        - Easy to use.
        - Example: Simple and user-friendly interface.
    - Scalability 
        - Can handle increased users.
    - Maintainability 
        - Easy to modify.
        - Easy to fix bugs.
        - Easy to upgrade.
    - Portability
        - Software should work on different platforms.
        - Example : Software runs on Windows, Linux, and macOS.

- Features
    - Describe system performance.
    - Improve user experience.
    - Define quality standards.
    - Cannot be observed directly as system functions

- Advantages
    - Improves software quality.
    - Increases customer satisfaction.
    - Makes software reliable.
    - Enhances security.
    - Improves performance.

---
### Difference Between Functional and Non-Functional Requirements

| Functional Requirement          | Non-Functional Requirement                       |
| ------------------------------- | ------------------------------------------------ |
| Describes what the system does. | Describes how the system performs.               |
| Defines features                | Defines quality attributes.                      |
| User-oriented.                  | System-oriented.                                 |
| Easy to test.                   | Measured using performance metrics.              |
| Example: Login System           | Example: Login should complete within 2 seconds. |

---

### Software Requirements Document (SRD) 
- Definition: 
    A Software Requirements Document (SRD) is a formal document that contains all the functional and non-functional requirements of the software system.

- Purpose
    - Acts as an agreement between the customer and the developer.
    - Guides software design, development, and testing.

- Contents of SRD
    - Introduction
    - Functional Requirements
    - Non-Functional Requirements
    - System Constraints
    - Assumptions
    - References

- Objectives of SRD
    - Clearly define software requirements.
    - Reduce misunderstandings.
    - Guide software development.
    - Help in testing and maintenance.
    - Improve communication among stakeholders.

---

### Software Requirements Specification (SRS)
- Definition:
    A Software Requirements Specification (SRS) is a detailed and structured document that describes all software requirements in a clear and precise manner.

- Characteristics of a Good SRS
    - Correct
    - Complete
    - Consistent
    - Verifiable
    - Modifiable
    - Traceable

- Importance
    - Helps developers understand requirements.
    - Reduces development errors.
    - Improves software quality.
    - Makes testing easier.
